Phase 1 - Business Understanding

1.1 Business Domain and Industry

Nimbasia ERP (internally also called FMS — Factory/Flow Management System) is a manufacturing ERP for Nimbasia Stabilizers, an Indian speciality-chemicals manufacturer producing metallic stearates and polymer additives (Zinc Stearate, Calcium Stearate, Magnesium Stearate, PE-Wax, GMS grades, PVC stabilizers such as CZ-100, lubricants such as Nimlub-T, etc.). The product catalogue in backend/src/config/masterData.js and the plant locations in frontend-hostinger/src/config/shipToLocations.js (Kota — Rajasthan, Bhiwandi & Khalapur — Maharashtra, Saykha — Gujarat) confirm the industry: batch chemical manufacturing with QC-gated release, GST-registered Indian legal entities.

1.2 Business Problem

Before this system, sales enquiries, order approvals, production batch cards, QC test sheets (in-process and finished goods), raw-material purchase orders, goods receipts, stock registers, packing and dispatch were tracked manually (registers/Excel/WhatsApp). This caused: lost enquiries and missed follow-ups, orders started without approval, no reliable link between what was produced, what passed QC, what was packed, and what was dispatched, untracked raw-material consumption per batch/shift, and no audit trail of who changed what.

1.3 Project Objective

One integrated web application that enforces the complete order-to-cash and procure-to-stock chain:

Enquiry -> Approval -> Sales Order -> Production (batches) -> QC (In-Process + Finished Goods) -> Inventory -> Packing -> Dispatch
Purchase Order -> (Pricing by Accounts) -> Sent to Supplier -> GRN + Incoming QC -> Raw Material Inventory -> Production Consumption

with role-based access, hard business gates (an order cannot dispatch more than has been packed; goods enter stock only after QC PASS; a GRN cannot be confirmed without a passed QC sheet), and a permanent audit log.

1.6 Scope, Assumptions, Constraints, Benefits

In scope

  • Sales: enquiries, approvals, orders, manual order requests, dashboard KPIs.
  • Purchasing: suppliers, purchase orders with price segregation, GRN with incoming QC, raw-material inventory.
  • Production: batch jobs, batch card wizard, in-process testing, finished-goods QC, batch substitution, BOM lookup.
  • Inventory: transaction ledger, stock register, adjustments, opening-stock import.
  • Packing and dispatch with partial shipments.
  • Administration: users (6 roles), master data (products, customers, suppliers, persons), activity log.

Explicitly out of scope (verified: no code exists)

  • Invoicing / accounting ledgers / payments (the "accounts" role only prices POs).
  • Email/SMS/WhatsApp/push notifications (UI toasts only).
  • File attachments, e-way bills, GST returns (a separate fuzzy.py GST reconciliation script exists in backend/ but is not wired to the app).
  • Multi-warehouse stock separation (warehouseLocation is a free-text label, not enforced).
  • Refresh tokens / password reset by email (see Security phase).

Assumptions

  • Single company, single database, trusted internal users on desktop and mobile browsers.
  • MySQL 8+ (raw SQL uses backticks, ON DUPLICATE KEY UPDATE, INFORMATION_SCHEMA).
  • One backend instance (in-memory response cache and master-data cache are per-process — see Performance phase).

Constraints

  • Quantities on Order/Production/Dispatch are integers (whole units); enquiry quantities may be fractional and are rounded up when converted to orders.
  • Production status may only be changed twice (MAX_PRODUCTION_STATUS_CHANGES = 2) and never after COMPLETED.
  • Enquiry approval is one-shot (only from PENDING).
  • GRN over-receipt tolerance is +2% of ordered quantity.

Benefits

  • End-to-end traceability from enquiry number to dispatched batch, including which raw-material batches went into each production batch.
  • QC-gated stock: nothing is dispatchable until a finished-goods QC sheet passes; nothing enters raw-material stock until GRN QC passes.
  • Segregation of duties in purchasing (requisition vs pricing vs receiving vs QC sign-off).
  • 30-day audit trail of every create/update/delete/approve with old and new values.

1.7 Real-World Use Case (end-to-end example)

  1. A customer phones asking for 500 KG Zinc Stearate. Sales user records an enquiry (mode "Phone", stage GENERAL). System assigns ENQ_0042.
  2. Admin/sales approves it on the Approval screen → order SO_0057 (group SO_012, order no ORD-0057) is created automatically, address auto-filled from the Customer Master.
  3. Sales fills city/pincode/state/country (mandatory gate) and clicks "Move to Production" → order becomes IN_PRODUCTION and a production batch is created (PENDING / "Not Started").
  4. Production supervisor opens the batch, fills the batch card (raw materials with batch numbers and shifts, additives, catalysts, equipment, process params). Setting status to Started (IN_PROGRESS) stamps productionStartedDate and deducts raw-material stock per line.
  5. A stearic-acid batch is found contaminated → supervisor performs a batch substitution; the ledger reverses the old batch and consumes the substitute, recorded permanently.
  6. QA records the in-process test sheet per shift; on completion of 500 units the batch reaches COMPLETED, and QA saves the Finished Goods Test Sheet with result PASS → 500 units enter finished-goods inventory and the order becomes READY_FOR_DISPATCH.
  7. Store packs 500 units using 20 "Zinc Stearate Trans. Bag" (packing-material stock deducted).
  8. Dispatch creates a shipment for 500 with status Dispatched → order becomes COMPLETED; finished-goods stock is written out.
  9. Every step above wrote an AuditLog row visible to admin on the Activity Log screen.

flowchart TD
  A[Batch PENDING 'Not Started'] -->|status change 1| B[IN_PROGRESS 'Started'
     productionStartedDate stamped

  B --> C{produced_quantity updates}

9.4 Packing → Dispatch

flowchart TD
  Q[Order READY_FOR_DISPATCH / PARTIALLY_DISPATCHED
    with a COMPLETED batch] --> PK[Pack: qty <= remaining AND <= FG stock;
    packing material qty <= its stock -> material OUT row]

    FG stock, packed - already dispatched)]
  DQ --> SS{Shipment status}
  SS -- DELIVERED only if it clears the remaining qty --> OK
  DQ --> SY[Order sync: 0 -> READY, partial -> PARTIALLY_DISPATCHED,
    full -> COMPLETED]
  DQ --> LG[FG OUT ledger row 'Dispatch #id' (delta-synced on edits)]

flowchart TD
  R[purchase/production raises PO DRAFT
    (no pricing fields accepted)] --> PR[accounts/admin prices items,

Insert / Update / Delete flows per table (summary)

  • Insert-then-finalize pattern: Order, PO, GRN and MOR-created orders are inserted with TMP-/TSO- placeholder numbers, then immediately updated with the id-derived business number — guarantees uniqueness without a counter table.
  • Replace-children pattern: all QC sheets and BOMs update via deleteMany + create of items inside the upsert.
  • Ledger discipline: InventoryTransaction is never updated/deleted; corrections are new ADJUSTMENT rows (reversal, delta sync, opening-stock delta).
  • Sample records: Enquiry {enquiryNumber:"ENQ_0042", companyName:"ABC Polymers", product:"ZINC STEARATE - ZS101 - 500 KG", status:"ACCEPTED"}; InventoryTransaction {type:"OUT", itemId:"Stearic Acid", quantity:120, batchNo:"SA-2607", shift:"A", reference:"Production #17"}.

Phase 25 - Performance Review

  • N-row scans for numbering (orders, MOR, supplier codes): O(all rows) per create — fine at thousands, degrade later (same fix as Bug #4).
  • Dispatch full dashboard loads all dispatchable orders + all dispatches + all orders for sales-group mapping in memory; the paginated mode still fetches everything then slices. Push filtering/pagination into SQL when volume grows.
  • Ledger groupBys are indexed and aggregate at the DB — good; the ledger only grows, so consider periodic snapshot/rollup rows after a few hundred thousand transactions.
  • getRawMaterialInventory loads every "IN" receipt to find last-received metadata — could be a MAX(createdAt) per item query.
  • Caching: 10–30s TTL in-memory caches with request coalescing on all hot lists; writes invalidate cross-module prefixes correctly. authMiddleware does one user SELECT per request (could cache briefly).
  • Frontend: all routes lazy-loaded (code splitting); VirtualizedTableBody windows large tables; Excel export is client-side (no server load); master data fetched once per session per 30s TTL. Bundle: xlsx is the heaviest dependency — consider dynamic import at export time.
  • No N+1 query patterns found (selects use nested Prisma select; GRN confirm loops per item inside one txn — acceptable for typical item counts).
  • Payloads: compression enabled >1 KB; selects.js keeps list payloads trimmed.

Phase 27 - Complete User Manual

27.1 Getting started (all users)

  1. Open the app URL (production: https://app.nimbasia.com). On mobile you can "Add to Home Screen" — it installs as an app (PWA).
  2. Sign in with the email and password your administrator gave you. [Screenshot: Login page]
  3. You land on your role's home screen. The left sidebar (☰ on mobile) shows only what your role can use. The top-right menu lets you change your password or log out.
  4. Tip: changing your password signs you out everywhere else automatically — that is intentional.
  5. Common mistake: more than 10 wrong passwords in 15 minutes locks logins from your network for a few minutes — wait, don't keep retrying.

27.2 Sales user guide

Recording an enquiry

  1. Sales → Enquiries → New Enquiry.
  2. Pick the enquiry date and mode (Phone/WhatsApp/…), choose the company (start typing — existing customers appear), and add one row per product: product (from the catalogue), grade, quantity, unit. [Screenshot: Enquiry form]
  3. Set the expected timeline (this later becomes the planned dispatch date), optional price + 3-letter currency, and the stage (General / Sampled / Quoted).
  4. Tick Urgent only for genuine rush jobs — the system will immediately create the order and a production job without waiting for approval.
  5. Warning: a product that isn't in the master catalogue will be rejected — ask an admin to add it under Master Data first.
  6. Tip: when you mark a stage as Sampled, the list shows a follow-up reminder after 12 days.

Approving / rejecting (admin or sales)

  1. Sales → Approval. Review the pending enquiry. This decision can be made exactly once.
  2. Approve → the sales order is created automatically with the customer's address pre-filled.
  3. Reject → you must type a reason (kept on record and audit-logged).

Orders

  1. Sales → Orders. Complete the delivery location (city, pincode, state, country) — production cannot start without it.
  2. Click Move to Production. After that, the order can no longer be edited or deleted.
  3. For an order without an enquiry, use the Manual Order Request flow; it must be approved, and dispatch sets its date to create the real order.
  4. FAQ: "Why can't I edit the order?" — production or a dispatch already exists against it; corrections happen downstream (batch card / dispatch records).

27.3 Production user guide

  1. Production → Production. Your queue is ordered urgent-first, then oldest-first. [Screenshot: Production queue]
  2. Open a batch → work through the steps: Batch Setup (batch no, capacity, personnel) → Raw Materials / Additives / Catalyst (pick items and batch numbers from stock — availability is shown; record the shift A/B/C) → Equipment → Process Params → Operation Log.
  3. Setting the batch to Started deducts raw-material stock. You may change the batch status at most twice, and never after Completed — plan the change.
  4. If a material batch is bad, use Substitute Batch on its row — never just retype the batch number; substitution returns the old stock, consumes the new, and keeps the traceability record.
  5. Record in-process test rows each shift (Production → In-Process Testing).
  6. Update Produced Quantity as output accumulates — reaching the order quantity completes the batch automatically.
  7. Save the QC Test Sheet with overall result. Only a Pass puts the goods into stock and makes the order dispatchable. The approver must be a different person from the analyst.
  8. Common mistake: completing the batch but forgetting the QC sheet — dispatch will see "No finished goods stock available".

27.4 Packing & Dispatch user guide

  1. Production/Dispatch → Packing: pick the order, enter packed quantity and the packing material + quantity used. The system blocks over-packing versus order or stock. [Screenshot: Packing]
  2. Dispatch → Dispatch: New Dispatch — quantity is capped at the least of (remaining order, stock, packed-but-not-dispatched). Date must be YYYY-MM-DD.
  3. Choose the shipment status: Packed / Dispatched / Delivered only when this shipment closes the whole remaining quantity.
  4. Dispatch → Dispatch Date: approved enquiries and manual requests that still need an order — enter the dispatch date to create the order.
  5. FAQ: "Dispatch says nothing is packed" — record packing first; packing is a tracked step, not a checkbox.

27.5 Purchasing guide (purchase / accounts)

  1. purchase role: Purchasing → Purchase Orders → New PO. Choose/enter the supplier (details auto-fill and update the register), the receiving plant (Ship To), and the items with quantities. You will not see price fields — that is by design.
  2. accounts role: open the draft PO, enter unit prices, tax %, currency, discount and freight, then Send to Supplier. This is your only lifecycle action; it is blocked until every item is priced.
  3. admin: when goods arrive, create the GRN against the PO, record received quantities (up to ordered +2%), complete the QC sheet (approver ≠ analyst), then Confirm — stock is posted and the PO progresses to Partially/Fully Received; Close it when done.
  4. Purchasing → Raw Materials shows live stock and the daily Stock Register; admins/production can post reasoned adjustments or import an opening stock take from Excel.

27.6 Administrator guide

  • Users: create accounts with one of six roles; reset passwords (the user is signed out everywhere).
  • Master Data: add products/persons/supervisors/modes/units/country codes; maintain Customer Master (codes CU-nnn) and Supplier Master (SO-nnn) with Excel import; deletes are soft (rows deactivate).
  • Activity Log: the audit trail of every action, searchable; kept 30 days.
  • Warnings: never run npm run seed against production (it wipes operational data); keep JWT_SECRET and CRON_SECRET private and long.

  2. Select shape in utils/selects.js; zod schemas in utils/validators.js.
  3. Service in src/services/xService.js (rules, transactions, audit, cache invalidation).
  4. Controller + route file; mount in app.js with authMiddleware + allowRoles.
  5. Frontend: page in src/pages, lazy route + role group in App.jsx, sidebar entry in config/navigation.js (update navigationRoles.test.js).

  • Missing tables (fresh DB) return empty lists by design (P2021 fallback) — run migrations if a list is unexpectedly empty.
